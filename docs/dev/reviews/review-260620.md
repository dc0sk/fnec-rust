# Project Review — 2026-06-20

## Scope and method

Full project-wide review covering code, tests, documentation, roadmap, backlog,
and configuration.  Based on `main` at commit `0f55bb6`.

---

## 1. Critical gaps

### GAP-001: `nec_worker` has zero tests despite 865 LOC

`crates/nec_worker/` contains 8 modules (865 lines of production code) but has
**0 unit tests** (`#[test]`) and **0 integration tests** inside the crate.
The only coverage is a single CLI integration test
(`apps/nec-cli/tests/worker_integration.rs`) that exercises the worker from
outside.

**Modules with no coverage:**
- `capability.rs` (104 LOC)
- `controller.rs` (82 LOC)
- `hosts.rs` (78 LOC)
- `protocol.rs` (102 LOC)
- `result_cache.rs` (181 LOC)
- `solve.rs` (140 LOC)
- `worker.rs` (150 LOC)

**Impact**: The cache module (`result_cache.rs`) and protocol serialisation are
unlikely to be exercised by the single CLI integration test.  Edge cases in
cache-key generation, capability inventory, and worker message round-trips are
untested.

**Action**: Add crate-level unit tests for each module, especially
`result_cache` (hit/miss/invalidation), `protocol` (serde round-trips, error
codes), and `hosts` (config parsing errors).

---

### GAP-002: `nec-tui` is a 6-line stub with no tests

`apps/nec-tui/src/main.rs` contains only:

```rust
fn main() {
    println!("fnec-tui — 4nec2-compatible antenna modeler (TUI)");
}
```

It has no tests directory and no dependency on `ratatui` or any TUI framework.
The Cargo.toml description says "terminal interface for batch runs and quick
inspection (ratatui)" but no such implementation exists.

**Impact**: Listed as a workspace member crate with no functional value.
Presents as a delivered feature but is entirely skeleton.

**Action**: Either implement the TUI (per roadmap) or remove it from the
workspace members list until development starts.

---

### GAP-003: `requirements.md` DEC-003 contradicts completed Phase 5

`docs/requirements.md` line 14 states:

> DEC-003: GPU acceleration starts in postprocessing only; matrix fill and
> solve acceleration are deferred.

This was the Phase 3-4 decision.  Phase 5 (PH5-CHK-007) completed full GPU
Hallen solve path (matrix fill + solve) and is marked Done.  The requirements
doc has not been updated to reflect this.

**Action**: Update DEC-003 to describe the current GPU capability scope
(wgpu-based matrix fill + solve, with stub fallback for CI).

---

### GAP-004: `requirements.md` DEC-010 describes outdated Hallen topology constraint

`docs/requirements.md` line 33 states:

> DEC-010: Hallen solve scope is explicitly constrained to collinear wire
> topologies aligned with the driven-segment axis; non-collinear Hallen
> requests must fail fast with a clear diagnostic.

Phase 2 completed the segmented hybrid Hallen reformulation, which supports
non-collinear and junctioned topologies via per-wire local cos(k·s) vectors
and KCL junction rows.  `--allow-noncollinear-hallen` is now a silent no-op.

**Action**: Update DEC-010 to reflect the non-collinear Hallen support status.

## 3. Test coverage gaps

### TEST-001: `nec-worker` unit tests — zero (see GAP-001)

### TEST-002: `nec-tui` — zero tests (see GAP-002)

### TEST-003: `nec-accel` GPU kernels are all stubs

Even though Phase 5 G7 "full GPU solve" is marked Done, all kernels in
`nec_accel` are CPU-emulation stubs.  The wgpu path is scaffolding only.
The `gpu_kernels.rs` module has 25 tests but they all test the CPU-emulation
path, not real GPU dispatch.

**Impact**: GPU acceleration cannot be measured on real hardware.  Performance
gates (G5, benchmark dashboard) exercise stub overhead only.

**Action**: Reconcile roadmap "Done" status with actual implementation state,
or add feature gates and CI hardware targets for real GPU execution.

### TEST-004: No fuzz targets run in CI

A `fuzz/` directory exists with a `nec_parser_parse` target and seed corpus,
but no CI workflow runs it.  Regression would be undetected until manual
invocation.

### TEST-005: `nec_project` has no unit tests in src/

`crates/nec_project/src/lib.rs` has 12 `#[test]` functions mixed into the
production source, but no dedicated test module or property-based tests.
The crate tests are all in `tests/project_roundtrip.rs`.

## 4. Backlog & roadmap discrepancies

### BR-001: PAR-011 status mismatch

`backlog.md` shows PAR-011 as `[ ]` (not done), but the last 16 commits
(0f55bb6 through 52d7244) all add PAR-011 test coverage.  The acceptance
criteria (drop-in profile detection, filename-steered matrix, contract tests)
are substantially met.

**Action**: Update backlog PAR-011 to reflect progress, or mark the specific
sub-items that remain (e.g., Windows-side replacement workflow).

### BR-002: Roadmap Phase 6 entry criteria ambiguous

`docs/roadmap.md` Phase 6 checklist items `PH6-CHK-005` through `PH6-CHK-007`
are all marked Done, but:
- PH6-CHK-005 (distributed execution design) — doc exists, no code.
- PH6-CHK-006 (SSH-backed worker deployment) — code exists but untested.
- PH6-CHK-007 (result cache) — code exists but untested.

The "Done" assessment appears to be doc-first rather than implementation-first,
contrary to the roadmap's "implementation target" column which lists code
artifacts.

### BR-003: CI benchmarking dashboard not operational

PH6-CHK-001 is marked Done, but the benchmark-dashboard.yml workflow and
benchmark artifact schema exist — there is no evidence of a live dashboard
publishing artifacts to gh-pages or a comparable endpoint.

## 5. Code quality observations

### CODE-001: 2 clippy warnings on MSRV incompatibility

```
warning: current MSRV is `1.75.0` but this item is stable since `1.82.0`
  --> apps/nec-cli/src/exec_profile.rs:18:28
```

`is_none_or` was stabilised in Rust 1.82.  The MSRV is pinned at 1.75.
Either bump MSRV or replace with a match/if-let expression.

### CODE-002: PT card parser vs model inconsistency

`docs/nec4-support.md` line 116 says PT card parser status is `unknown`, but
`nec_model` has a `PtCard` struct (referenced in `nec_parser`'s imports? no —
PT is not imported in `crates/nec_parser/src/lib.rs`).  The `warn_pt_card`
function exists in the CLI but the parser doesn't store PT cards.

This is a known gap documented in the support matrix ("warning function defined
but not yet called from main"), but it creates a confusing state where the
model supports it but the parser doesn't.

### CODE-003: GM/GR parser gap remains open

Geometry builder (`nec_solver::geometry`) supports GM rotate/translate and GR
z-axis rotation repeat, but the parser does not recognise GM or GR cards.
Decks with these cards produce an unknown-card warning rather than applying
the transform.  Marked in the support matrix as a Phase 3 item, but Phase 4 is
already complete.
