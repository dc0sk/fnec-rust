# Contributing

## Overview

fnec-rust is a Rust-native NEC-compatible solver with CLI, GUI, and TUI frontends. See
[docs/architecture.md](architecture.md) for the system structure and crate responsibilities, and
[docs/design.md](design.md) for product and UX direction.

## Prerequisites

- Rust stable toolchain (edition 2021). Install via [rustup](https://rustup.rs/).
- `cargo` in `PATH`.
- `gh` CLI for pull-request and CI operations (optional but expected for contributors with commit access).
- Linux x86-64 is the primary development target; ARM64 is tested in CI.

## Build

```
cargo build
```

Release build:

```
cargo build --release
```

The workspace produces three binaries under `target/{debug,release}/`:

| Binary   | Source crate  | Description                     |
|:---------|:--------------|:--------------------------------|
| `fnec`   | `apps/nec-cli` | CLI solver — primary entry point |
| `fnec-gui` | `apps/nec-gui` | GUI (iced)                     |
| `fnec-tui` | `apps/nec-tui` | TUI (ratatui)                  |

## Test

Run the full test suite:

```
cargo test
```

Run only a specific crate:

```
cargo test -p nec_solver
```

Run only a specific integration test file:

```
cargo test -p nec-cli --test corpus_validation
```

Run a single test by name:

```
cargo test -p nec-cli --test corpus_validation par001
```

### Corpus and CI gates

The `apps/nec-cli/tests/` directory contains the CI integration gates:

| File | Purpose |
|:-----|:--------|
| `corpus_deck_sanity.rs` | Parses every corpus deck without panicking |
| `corpus_validation.rs` | Numerical tolerance gates + documentation contracts |
| `ground_diagnostics.rs` | Ground-model fail-fast and diagnostic coverage |
| `parser_warnings.rs` | Parser warning paths |
| `report_contract.rs` | Report output format and section ordering |

The `corpus/` directory holds the reference NEC decks and `corpus/reference-results.json`, which
is the authoritative tolerance contract for numerical results. Any change that shifts a reference
value must update `corpus/reference-results.json` with the new expected value and include a clear
justification in the PR description.

Do not replace reference values generated by fnec-rust itself as a proxy for external NEC2
validation. Changes to numerical reference values require external basis (e.g. comparison against
a published NEC-5 validation manual scenario, measurement data, or analytical ground truth).

## Pre-push sequence

Before every `git push`, run:

```
cargo fmt
cargo check
cargo test
```

All three must pass with no errors. Warnings in non-test code are acceptable during active
development but should not be introduced deliberately. Push only if all three pass cleanly.

**Do not bypass CI** with `--no-verify` or equivalent. Do not amend published commits on shared
branches.

## Branch conventions

| Branch pattern | Purpose |
|:---------------|:--------|
| `main` | Protected. Never commit directly to `main`. |
| `feat/<slug>` | New features or improvements |
| `fix/<slug>` | Bug fixes |
| `docs/<slug>` | Documentation-only changes |
| `refactor/<slug>` | Code restructuring without behaviour change |

Use `ph3-chk-NNN` slugs for roadmap checklist items (e.g. `feat/ph3-chk-003-plugin-api`).

Keep branches short-lived. Open a PR as soon as the work is ready for review, even if rough.

## Pull request process

1. Create a feature branch (never work on `main` directly).
2. Run the pre-push sequence and confirm it is clean.
3. Push and open a PR via `gh pr create` with a descriptive title and body.
4. All 10 CI checks must pass before merge. Do not merge with failing CI.
5. Merge strategy: **squash-merge only**. Delete the branch after merge.
6. Update `docs/changelog.md` in the same PR for any user-visible change.

## Corpus-gate requirements

A corpus gate is an integration test in `corpus_validation.rs` that:

- Reads a `.nec` deck from `corpus/`.
- Runs the solver end-to-end.
- Compares a key result (impedance Re/Im, peak gain, sweep point count) against a value in
  `corpus/reference-results.json` within a stated tolerance.

New corpus gates must:

1. Add a deck to `corpus/` if the scenario is not already covered.
2. Add the reference value(s) to `corpus/reference-results.json`.
3. Add a test function to `corpus_validation.rs` with a descriptive name.
4. Pass on every supported platform (Linux x86-64 minimum; ARM64 preferred).

Documentation contract tests (`par*` prefix) live in the same file but do not require a corpus
deck — they validate that documentation files contain required content.

## Documentation

All files under `docs/` must carry the standard YAML frontmatter block:

```yaml
---
project: fnec-rust
doc: docs/<filename>.md
status: living
last_updated: YYYY-MM-DD
---
```

The CI `validate-doc-frontmatter` job checks every `docs/*.md` file. A missing or malformed
frontmatter block will fail CI. The `last_updated` field is updated by the
`scripts/stamp-doc-last-updated.sh` helper when the CI `stamp-docs` job runs on merge to `main`.

Do not hand-edit `last_updated` on `main`; do update it when working on a docs file on a feature
branch so reviewers see the correct date.

## Architecture orientation

For new contributors the recommended reading order is:

1. [docs/architecture.md](architecture.md) — crate graph, pipeline stages, and compatibility model.
2. [docs/design.md](design.md) — product and UX direction.
3. [docs/nec4-support.md](nec4-support.md) — which NEC cards are recognized and what their
   current status is (parser vs. functional).
4. [docs/requirements.md](requirements.md) — formal requirement tags referenced in code comments.
5. `crates/nec_parser/src/lib.rs` — the entry point for deck parsing.
6. `crates/nec_solver/src/lib.rs` — the top-level solver dispatch.
7. `apps/nec-cli/src/main.rs` → `solve_session.rs` — the CLI execution path.
